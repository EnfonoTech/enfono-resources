Project Delivery — Standard Operating Procedure

SOP-ERP-002 · Version 2.0 · Internal Use Only. The single operating reference for every Enfono project — from signed contract to go-live to annual maintenance — across both Own and Collaborated delivery models. Owner: Project Manager + Functional Consultant. Reviewed annually or after any major project retrospective.

Owner tags used throughout: PM Project Manager   FC Functional Consultant   TL Technical Lead   SH Sales Head   PM+FC both

1. Objective

Deliver every Enfono project in a way that is consistent, on time, on budget, and easy for any team member to follow — whether Enfono owns the client relationship directly (Own Project) or delivers through a partner (Collaborated Project). This SOP exists to:

  • Make responsibilities unambiguous at every step — every action carries an owner tag.
  • Protect Enfono and the client from scope creep through written specifications and Change Requests.
  • Keep the client always informed — a weekly report plus a client meeting every three days.
  • Run the delivery team on a predictable rhythm — a daily standup and a weekly project plan.
  • Standardise good delivery — improving efficiency, consistency, customer experience, scalability, and profitability.

6. Step-by-Step Workflow

Stage 1 — Sales & Handover (Day 0)

Trigger: signed contract received. Sales hands over to PM within 24 hours; PM briefs FC the same day.

  • SH→PM Hand over signed contract / SOW — scope, value, payment milestones, client contacts (POC, IT, Accountant).
  • PM Confirm delivery model: Own or Collaborated.
  • PM Note any verbal commitments outside the SOW — log as risks; never commit verbally.
  • FC Read the SOW in full; flag ambiguous requirements or gaps before kickoff.

Stage 2 — Implementation

  1. Team Preparation Meeting (within 24 hrs)

    PM+SH prepare the Implementation BRD (scope, deliverables, explicit out-of-scope list, expectations). PM creates the Project Charter, sets up the tracker, shared folder, Custom Feature Register, and the project-wise Teams group. FC shares data templates + drafts the kickoff agenda.

  2. Kickoff Meeting with Client (Days 1–3)

    PM+FC facilitate: scope walkthrough → review the out-of-scope list aloud and get acknowledgement → timeline + data deadlines → custom feature deep-dive (assign a Functional Champion each) → client responsibilities → communication plan. PM sends minutes within 24 hrs and obtains written POC acknowledgement.

The five delivery phases — each ends with a milestone the client signs off before the next begins.

Phase 1 — Foundation & System Setup (Wk 1–2) Standard

  • TL Provision server (DNS, SSL, firewall, SSH); install Frappe v15 + ERPNext; set up site + admin.
  • FC Company setup, localised Chart of Accounts + cost centers, users/roles/permissions, warehouses, SMTP tested.
✓ Milestone: all users log in, email works, COA in place. FC demos to POC; PM gets sign-off.

Phase 2 — Core Module Configuration (Wk 3–4) Standard + Custom

  • FC Import master data; naming series + standard print formats; core workflows (Sales SO→DN→SI, Purchase PO→GRN→PI, stock); demo to Champions.
  • FC Custom Stage 1 — Requirement Capture (AS-IS → TO-BE, data, output, edge cases; log in register).
  • FC Custom Stage 2 — Feature Specification (objective, scope in/out, flow, fields, logic, acceptance criteria, estimate).
  • PM Approve spec against SOW; allow 2 business days for written client approval — no development before sign-off.
  • TL Custom Stage 3 — Dev Estimate (+20% buffer); if >20% over SOW → raise a Change Request.
✓ Milestone: core transaction cycle tested end-to-end; all Phase 2 specs approved in writing.

Phase 3 — Accounts, Finance & Approvals (Wk 5–6) Standard + Custom

  • FC Verify GL entries; tax templates (VAT/GST), P&L, Balance Sheet; approval chains.
  • TL Import Opening Trial Balance — only after client accountant sign-off.
  • TL Custom Stage 4 — Build & Unit Test: dedicated app, no hardcoded credentials, no raw SQL, integrations logged + retry + idempotent, unit tests on all paths.
✓ Milestone: accounting live, first JE posted, approval cycle tested, OTB imported. PM + Accountant sign off.

Phase 4 — Reports, Outputs & Print Formats (Wk 7–8) Standard + Custom

  • FC Standard reports, branded Customer SOA, dashboards, role-specific user guide; print-format review call.
  • TL Custom Stage 5 — Staging Review: deploy to staging, bench migrate, performance check at realistic volumes.
  • PM Sign off staging before any custom feature is shown to the client.
✓ Milestone: all reports run, print formats signed off, custom features pass staging review.

Phase 5 — UAT, Data Migration & Go-Live (Wk 9+) Standard + Custom

  • TL Final data migration as of cut-off; PM+FC freeze legacy + communicate the cut-off.
  • FC Custom Stage 6 — UAT: test script per feature; Champions execute; record Pass/Fail.
  • PM Triage by severity; obtain written sign-off per feature once Critical/Major resolved.
  • FC Training (operations + finance/manager); confirm users work independently.
  • PM Custom Stage 7 — Production Deploy: DB backup + rollback plan, off-peak, smoke-test, monitor 30 min.
UAT SeverityDefinitionFix target
CriticalBroken / wrong GL or stock entry24 hrs — retest before proceeding
MajorKey criterion fails, no workaround48 hrs — retest
MinorCosmetic / non-blockingPost-go-live or next sprint
✓ Milestone: production go-live. PM collects the signed Go-Live Acceptance Form from the Business POC.

Stage 3 — Communication & Finance (runs throughout)

  • PM Own Project — PM handles all client communication, invoicing, and collection directly per billing milestones.
  • FC Collaborated Project — the partner owns communication + payment; FC keeps the partner updated; the partner chases delayed responses.
  • PM Invoice on milestone sign-off; follow up on overdue payments per contract.

Stage 4 — Completion & AMC

  • Hypercare (Wk 10–12): FC daily 15-min check-in first live week; PM+FC respond to Critical issues within 2 business hours; new requests → CR.
  • Project Close: PM+FC post-implementation review; PM archives docs, client signs final delivery acceptance, final invoice; FC writes the support handover note.
  • AMC: transitions into an Annual Maintenance Contract for ongoing support.
AMC scope, duration, SLAs, and commercials are defined in the maintenance agreement signed at completion. Work outside AMC scope is handled as a separate Change Request.

8. Change Requests — the Scope Gate

1 · Identify▶ 2 · Log in CR Register▶ 3 · Assess impact / effort / cost▶ 4 · Written CR to POC▶ 5 · POC approves in writing▶ 6 · Update plan / invoice
Hard rule: no work on any Change Request begins until written client approval is received.

9. Defining Out of Scope

Before kickoff, the PM and FC produce a written Out-of-Scope list inside the BRD. For every major area, state explicitly what is NOT included — e.g. "Payroll module out of scope", "Migration limited to open balances only", "Third-party integrations not listed are out of scope".

The rule: anything not written as in-scope is out-of-scope by default. Any later request for an out-of-scope item follows the Change Request process — no exceptions, no verbal agreements.

10. Quality Checklist

Before each phase sign-off

  • Phase milestone criteria met and demoed to the client.
  • Client sign-off obtained in writing before starting the next phase.
  • Tracker and Custom Feature Register updated.

Before any custom feature is built

  • Signed Feature Specification exists (objective, scope in/out, acceptance criteria).
  • Effort estimated; if >20% over SOW → CR raised and approved.

Before any custom code reaches production

  • Staging test passed against acceptance criteria.
  • PM sign-off recorded; DB backup taken; rollback plan documented.
  • Code in dedicated app; no hardcoded credentials; no raw SQL; external calls wrapped with retry/idempotency.

Before go-live

  • All UAT Critical + Major issues resolved and retested.
  • Written UAT sign-off per feature from Functional Champion.
  • Opening Trial Balance signed off by client accountant before import.
  • Users trained and confirmed able to work independently.
  • Go-Live Acceptance Form signed by Business POC.

13. Common Mistakes to Avoid

  • Committing verbally — log it and route to a CR instead.
  • Building a custom feature with no signed spec. No spec, no code.
  • Skipping the written out-of-scope list at kickoff.
  • Importing the Opening Trial Balance without accountant sign-off.
  • Deploying custom code straight to production (no staging / backup / rollback).
  • Letting client data slip silently — chase weekly, escalate to POC if >3 days late.
  • Going quiet — the weekly report and 3-day client meeting are non-negotiable.
  • Hardcoding credentials or using raw SQL in custom code.
  • Starting CR work before written client approval.
  • Closing the project without the signed Go-Live Acceptance Form and a support handover note.
  • Treating a Collaborated project like an Own one — keep the partner in the loop.

14. Weekly & Monthly Rhythm

Two clocks: the weekly clock drives delivery toward the next phase gate; the monthly clock protects budget, scope, risk, and the client relationship.

Plan the week (Monday, PM + FC): plan backwards from the current phase's milestone — each week is one slice toward that gate. Anchor (phase + gate date + weeks left) → pull the slice (what must close this week) → capacity (team availability; standup covers 2 projects) → commit (owners, due dates, dependencies) → client deliverables (send anything needing the POC on Monday, not Thursday). Run the Weekly Project Run Checklist each week.
