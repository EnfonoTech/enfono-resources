Docs Hub & GitHub Webhooks

Docs Hub is Enfono's static documentation hosting platform. Every docs site is a Git repository — push to the configured branch and your changes go live automatically. No FTP, no manual deploys, no server access needed.

Architecture

GitHub Push
  │
  ▼
POST https://docs.enfonoerp.com/webhook  (HMAC-SHA256 verified)
  │
  ▼
docs-hub-agent  (port 3849 on docs server)
  │
  ├─ git pull (on matching repo in /opt/docs-hub/sites/)
  └─ caddy reload  (picks up new static files)
  │
  ▼
Live at: https://{site-name}.docs.enfonoerp.com

Key Components

1. Docs Server

  • Caddy: Reverse proxy + automatic HTTPS via Cloudflare DNS-01 challenge
  • docs-hub-agent: Hono server on port 3849 — handles deploy, delete, and webhook endpoints
  • Site storage: /opt/docs-hub/sites/{site-name}/ — bare Git repos with working trees
  • SSL: Wildcard cert for *.docs.enfonoerp.com

2. Control Server (Server Manager)

  • API: manager.enfonoerp.com/api/docs-hub/sites — CRUD for docs sites
  • Database: SQLite tracking site metadata, deploy status, webhook configs
  • Agent proxy: Forwards deploy/delete calls to docs-hub-agent on the docs server
  • Cloudflare DNS: Auto-creates A records pointing {site}.docs.enfonoerp.com to docs server IP

3. Caddy Routing

Each site gets a Caddy handle block. The agent manages these via /opt/docs-hub/Caddyfile.sites:

# Auto-managed by docs-hub-agent
# Site: rmax-docs → /opt/docs-hub/sites/rmax-docs

rmax-docs.docs.enfonoerp.com {
    root * /opt/docs-hub/sites/rmax-docs
    file_server
    encode gzip
}

Auto-Deploy via GitHub Webhooks

How it works

  1. GitHub sends a POST to the webhook URL

    Every push to the configured branch triggers a push event. GitHub POSTs a JSON payload to the webhook endpoint.

  2. docs-hub-agent verifies the HMAC-SHA256 signature

    The agent computes sha256(hmac, payload) and compares it against the X-Hub-Signature-256 header. Rejects unverified requests.

  3. Agent matches the repository name from the payload

    Extracts repository.full_name (e.g., EnfonoTech/rmax-docs) and maps it to a configured site.

  4. git pull on the matching site directory

    Runs git pull origin {branch} inside /opt/docs-hub/sites/{site-name}/.

  5. Caddy reloads configuration

    Static files are refreshed. New content is served immediately.

Webhook URL

Every Docs Hub site uses the same webhook endpoint:

https://docs.enfonoerp.com/webhook

GitHub delivers the repository name in the payload, so one URL handles all sites.

Setting up a webhook in GitHub

  1. Go to your docs repository on GitHub

    Navigate to Settings → Webhooks → Add webhook.

  2. Configure the webhook

    Payload URLhttps://docs.enfonoerp.com/webhook
    Content typeapplication/json
    SecretAsk the admin for the webhook secret
    SSL verificationEnable
    EventsJust the push event

  3. Save and test

    After saving, GitHub sends a test ping. Check the "Recent Deliveries" tab for the response. A successful ping returns HTTP 200.

Important: The webhook secret must match the WEBHOOK_SECRET environment variable on the docs server. Without it, HMAC verification fails and deploys won't trigger.

Webhook Security

The webhook endpoint implements three layers of verification:

LayerMechanismWhat it prevents
1. HMAC-SHA256 X-Hub-Signature-256 header verified against shared secret Forged payloads, replay attacks from non-GitHub sources
2. Event type filter Only push events are processed Unwanted triggers from PRs, issues, releases
3. Repository mapping Payload repository.full_name must match a registered site Deploys to non-existent or wrong sites

Verification flow (agent-side)

// docs-hub-agent: webhook verification
function verifySignature(payload: string, signature: string): boolean {
  if (!signature || !WEBHOOK_SECRET) return false;

  const computed = crypto
    .createHmac('sha256', WEBHOOK_SECRET)
    .update(payload)
    .digest('hex');

  const expected = `sha256=${computed}`;
  return crypto.timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(expected)
  );
}

DNS and Domain Setup

When you create a new docs site, the system automatically:

  1. Creates a Cloudflare DNS A record pointing {site-name}.docs.enfonoerp.com to the docs server
  2. Registers a Caddy route in /opt/docs-hub/Caddyfile.sites
  3. Sets up the Git repo on the docs server
  4. Performs the initial deploy (git clone + Caddy reload)

The wildcard SSL certificate (*.docs.enfonoerp.com) covers all subdomains — no per-site cert setup needed.

DNS propagation: New DNS records can take 1-5 minutes to propagate. If your site returns an NXDOMAIN error right after creation, wait a minute and try again.

Troubleshooting

Site not updating after push

  1. Check GitHub webhook "Recent Deliveries" — does it show HTTP 200?
  2. If HTTP 500: check docs-hub-agent logs: ssh docs-server 'journalctl -u docs-hub-agent -n 50'
  3. If HTTP 401: webhook secret mismatch. Verify WEBHOOK_SECRET on server matches GitHub.
  4. If HTTP 200 but no update: the agent may have failed to match repository.full_name. Check the site's gitRepo in Server Manager matches exactly.

DNS not resolving

dig {site-name}.docs.enfonoerp.com +short

Should return the docs server IP. If not, check Cloudflare DNS records in the dashboard.

SSL certificate error

The wildcard cert uses Cloudflare DNS-01 challenge. If Caddy fails to obtain/renew:

# On docs server:
ssh docs-server
systemctl status caddy
journalctl -u caddy -n 50 | grep -i "acme\|tls\|cert"
Pro tip: Use the "Webhook Guide" button on any site card in Server Manager for a copy-pasteable setup walkthrough including the exact webhook URL and secret.
